# 14 — AI Workflow

This document specifies the end-to-end AI reasoning workflow — what happens conceptually at each pipeline stage, independent of which specific agent implements it (`15_Agent_Workflow.md`) or which API endpoint triggers it (`13_API_Specification.md`). Read this as "the thinking process," the other two as "who does the thinking" and "how it's invoked."

## 14.1 Design Principle: Staged, Not Single-Shot

> **Decision:** No stage is generated by a single unstructured prompt that tries to produce the whole artifact in one pass. Every stage follows a **decompose → generate → cross-check → assemble** internal sequence. Reasoning: single-shot generation of a structured artifact (a schema, an API contract) is exactly where LLMs are most prone to internal inconsistency (e.g., inventing a field in the diagram that doesn't exist in the DDL). Decomposing forces intermediate structured representations that can be validated before assembly, which is the same reason `12_Database_Design.md` stores `entities`/`relationships` as the source of truth and *derives* `raw_ddl` and the ERD from it, rather than generating three independent outputs that could drift.

## 14.2 Stage-by-Stage Workflow

### Stage 1 — AI Interview
**Input:** raw user idea (free text) + prior answers.
**Process:** classify which of the four required fields (problem, target user, value prop, constraints — FR-101) are still unresolved or ambiguous → generate at most one targeted clarifying question per turn (never a batch of five at once, to keep the interaction conversational per `09_User_Flow.md §9.2.1`) → detect contradictions against already-captured answers (FR-102).
**Output:** structured `interview_data` object.
**Cross-check:** none yet — this is the root of the dependency graph (`12_Database_Design.md §12.2` note on `artifact_versions`).

### Stage 2 — Idea Validation
**Input:** `interview_data`.
**Process:** score three dimensions independently (technical feasibility, market signal, scope realism — FR-111), each dimension reasoned about separately rather than one holistic "vibe" score, then combined into the scorecard. Scope realism specifically cross-checks feature ambition against the stated constraints field.
**Output:** validation scorecard + flags.
**Cross-check:** flags are advisory only (FR-113) — no blocking gate.

### Stage 3 — Market / Competitor Analysis (optional)
**Input:** `interview_data` + user-supplied competitor names.
**Process:** structure known competitor claims separately from any live-fetched claims, tagging provenance per claim (FR-121/122) — this tagging happens at generation time, not as a post-hoc label, so it can never be silently dropped downstream.
**Output:** competitor comparison table + directional market notes.

### Stage 4 — PRD Generation
**Input:** `interview_data` + validation + (optional) market analysis.
**Process:** generate section-by-section (problem, goals, scope, features, success criteria — FR-132) rather than one long-form document pass, so the scoped "regenerate this section" UX (`09_User_Flow.md §9.2.4`) is a real capability, not a UI illusion over a monolithic regeneration.
**Output:** `prds.sections` (structured, section-keyed).

### Stage 5 — BRD Generation
**Input:** PRD (by reference, FR-142).
**Process:** generate content that is *derived from but distinct from* the PRD — business justification, stakeholder/ROI framing — explicitly instructed to avoid restating PRD scope/features verbatim (enforces FR-141 at the generation-prompt level, not just a post-hoc check).

### Stage 6 — User Stories + Acceptance Criteria
**Input:** PRD + BRD.
**Process:** enumerate stories per persona-in-project, each tagged to a specific PRD feature/goal it satisfies (traceability requirement underlying FR-151); generate acceptance criteria as a second, dependent pass per story rather than inline with the story itself, so criteria quality doesn't get diluted by also trying to phrase the story well in the same generation call.
**Output:** stories + linked criteria; each story enters `draft` status until a human marks it `ready` (FR-152 gate is human-in-the-loop by design, not automatic).

### Stage 7 — System Architecture
**Input:** Stories (only `ready` ones).
**Process:** identify components implied by the stories' actions, group into a service/component diagram, distinguish clearly between "core" components (implied directly by scope) vs. "supporting" components (auth, billing, notifications) that are near-universal — flagged as such so the user understands what's boilerplate vs. what's bespoke to their idea.
**Output:** Mermaid component diagram + component list.

### Stage 8 — Database Design
**Input:** Stories + Architecture.
**Process:** entity extraction pass (nouns/objects implied by stories) → relationship inference pass (verbs/associations) → normalization check pass (explicit sub-step looking for repeating groups / missing FKs, FR-163) → only then assemble into `entities`/`relationships` JSONB. See sequence diagram in `11_System_Architecture.md §11.5`.
**Output:** schema JSONB + derived ERD + derived DDL.

### Stage 9 — API Design
**Input:** Database Design + Stories (for action verbs → HTTP methods mapping).
**Process:** for each entity, derive CRUD endpoints where implied by stories; for each *non*-CRUD story action (e.g., "checkout," "invite collaborator"), derive a purpose-built endpoint rather than forcing it into generic CRUD — this distinction is made explicitly during generation, not left to chance.
**Output:** endpoint list (FR-172 shape).

### Stage 10 — Task Breakdown
**Input:** Architecture + API Design + Schema.
**Process:** group by component (per Architecture), then by entity/endpoint touched, then sequence tasks respecting technical dependency order (e.g., schema migration tasks before endpoint-implementation tasks) — this ordering logic is itself a generation sub-step, not left to arbitrary list order.
**Output:** milestone-grouped task list (FR-191 shape).

### Stage 11 — Prompt Generation
**Input:** a single task + its linked schema/API/criteria fragments.
**Process:** assemble (not re-generate) a self-contained prompt embedding the exact upstream fragments verbatim from stored artifacts, plus task-specific instructions — deliberately the *least* generative stage in the pipeline, since correctness here depends on faithful embedding of already-validated upstream content, not fresh reasoning (see `16_Prompt_Workflow.md` for the full prompt template).
**Output:** Markdown/JSON prompt (FR-192/193).

## 14.3 Cross-Cutting: The Reviewer Pass

After Stages 6 through 10 specifically (the stages with the most cross-artifact consistency risk), a Reviewer pass runs before the artifact is presented as "complete" (not merely as an optional on-demand action) — this is the automatic half of FR-211, distinct from the on-demand AI Product Critic (FR-212) detailed in `15_Agent_Workflow.md`. The Reviewer pass checks specifically for:

- References to entities/endpoints that don't exist in the upstream artifact.
- Stories with no corresponding architecture component or task.
- Acceptance criteria that contradict the API contract's stated behavior.

## 14.4 Regeneration vs. Fresh Generation

When a user re-syncs a stale artifact (`11_System_Architecture.md §11.6`), the workflow is **not** identical to first-time generation: regeneration is instructed to preserve any part of the current artifact that isn't affected by the specific upstream change (identified via the `stale_reason` field), rather than regenerating the whole artifact from scratch. This matters concretely: editing one user story should not silently rewrite unrelated parts of an already-good database schema.
